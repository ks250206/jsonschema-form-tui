# AGENT_rust.md

## 目的

このプロジェクトは、**JSON Schema に準拠した JSON を素早く作成・編集できる TUI アプリケーション**を Rust で構築するためのものです。  
目指すプロダクトは、ひとことで言うと **「react-json-schema-form の lazygit 風 TUI 版を Rust + ratatui で実装したもの」** です。

主な目的は以下です。

- JSON Schema からフォームを生成する
- ユーザーが TUI 上で JSON を安全に入力できる
- 入力結果が JSON Schema 2020-12 に従って検証される
- lazygit のようなペイン分割 UI と VIM ライクな操作感を提供する
- AI コーディングエージェントが迷いにくいよう、実装方針・制約・優先順位を明文化する

---

## プロダクト要約

このアプリは、JSON Schema を入力として読み込み、対応するフォームを TUI 上に構築し、ユーザーが JSON を編集・生成・保存できるようにするツールです。

### 中核となる利用シーン

1. 開発者が JSON Schema ファイルを読み込む
2. TUI がフォームを生成する
3. ユーザーがフォームに値を入力する
4. 入力値は onBlur 相当のタイミングを基本として検証される
5. 生成された JSON が常に確認できる
6. 必要に応じて jq 風フィルターで出力 JSON を絞り込める
7. JSON を指定パスへ保存できる

## TUI の基本要件

### UI コンセプト

- **lazygit 風**のペイン分割 UI
- **キーボード主体**
- **VIM ライク**な操作体系
- ただしフォーム入力では **Tab 移動**、**マウスクリック**も許容する
- ログ、出力、スキーマ、フォームが同時に把握しやすいこと

### 現在の実装で確定したレイアウト

Rust 版でも `edit` モードでは以下の固定レイアウトを基準とする。

- 最上段 full-width: **Schema Path ペイン**
- 左上: **Schema ペイン**
- 左下: **Form ペイン**
- 右上: **Filter ペイン** (縦幅数行の小さいペイン)
- 右下: **Output ペイン**(縦幅大きめ)
- 下部 full-width: **Log ペイン**
- 最下段 full-width: **Footer**

各ペインタイトルには `[1]` から `[6]` の番号を表示し、数字キーで即時フォーカス移動する。
マウスクリックでもペインフォーカスを変更できる。
アクティブペインは lazygit 風に枠線色を変えて示す。

### カーソル方針

- 各ペインは独立したカーソル座標を持つ
- ハイライトは**行全体ではなく1セル単位**で行う
- `hjkl` はアクティブペイン内カーソル移動の責務を持つ
- `1-6` とクリックがペインフォーカス変更の責務を持つ

### 編集可能ペイン

Rust 版でも、少なくとも以下のペインは TUI 上で直接編集できるようにする。

- `[1] Schema Path`
  - 1行目: 読み込むスキーマ識別子またはファイルパス
  - 2行目: 保存先 JSON パス
- `[2] Schema`
  - JSON Schema 本文を行単位で編集
- `[3] Form`
  - フォーム値をフィールド単位で編集
- `[4] Filter`
  - jq 風フィルター文字列を編集

`[5] Output` と `[6] Log` は読み取り専用でよい。

### Schema Path ペインの知見

- スキーマ切替を高速化するため、**bundled schema** を複数内蔵する
- 少なくとも `sample/basic`, `sample/profile`, `sample/deploy` 相当は持つ
- `[1]` の1行目に bundled schema 名を入力して切り替えられること
- 存在しない bundled 名の場合は、ファイルパスとして読み込みを試みること

### Schema ペイン編集の知見

- `[2] Schema` は行単位編集を行う
- 編集結果は即時に `schema_text` 相当へ反映してよい
- ただし、**JSON として再パース成功した場合だけ** schema / form / output を再構築する
- パース失敗時は表示テキストを保持しつつ、ログに明示的なエラーを出す

## モード設計

このアプリではモードを明確に分けること。  
少なくとも以下の 2 層を持つ。

1. **画面モード**
2. **入力モード**

### 1. 画面モード

#### edit モード

レイアウト:

- 上部 full-width: **JSON Schema ファイルパスのペイン**
- 左上: **JSON Schema ペイン**
- 左下: **フォームペイン**
- 右上: **JSON 出力フィルターペイン**
- 右下: **JSON 出力ペイン**
- 下部 full-width: **ログペイン**
- 最下段 full-width: **操作説明フッター**

#### form モード

基本構成は edit モードに近いが、以下を満たすこと。

- **JSON Schema ペインは非表示または強く縮退**
- **フォームペインを主役として押し出す**
- **JSON 出力パスのペインを追加または明示**
- 出力 JSON は確認しやすく保つ
- ログとフッターは維持する

### 2. 入力モード

入力モードは最低限以下を分離すること。

- **normal**: 全体操作
- **insert**: テキスト入力
- **visual**: 選択・コピー用
- **field-focus**: フォーム部品にフォーカスして操作する状態

ただし v1 では、必要以上に複雑なモード爆発を避けること。

## 操作要件

### グローバル操作

基本は VIM 準拠とする。

- `h j k l`: アクティブペイン内カーソル移動
- `0` / `$`: 行頭・行末へ移動
- `gg` / `G`: 先頭・末尾へ移動
- `w` `W` `b` `B`: 単語単位移動
- `a` `i`: 挿入系の遷移
- `v`: visual モード遷移
- 数字でペイン移動
- `Tab` / `Shift+Tab`: フォーム要素移動
- マウスクリックでフォーカス変更可能

### 現在の削除系操作

Rust 版でも、normal モードの削除系は以下を基準とする。

- `dd`: 現在の編集対象行を空にする
- `D`: カーソル位置から行末まで削除
- `dw` / `dW`: カーソル位置から次単語境界まで削除
- `db` / `dB`: カーソル位置から前単語境界まで削除
- `diw` / `diW`: inner word / inner WORD を削除
- `daw` / `daW`: outer word / outer WORD を削除

現時点では以下は**対応しない**。

- `5dl` のような count 付き削除
- 数字プレフィックス付き motion

理由:

- 数字キーはペインフォーカス移動に即時に使う
- 削除 count と数字ペイン移動を両立すると体感遅延が出やすい
- 現状は**数字キー即応性を優先**する

### 優先順位

同じキーに複数の意味がありうる場合は、以下の優先順位を守る。

1. 入力中のテキストフィールドのローカル編集
2. フォーム部品の局所操作
3. 全体のペイン操作

### フッター

フッターには常に最低限以下を表示すること。

- 現在モード
- 主なキー操作
- 保存方法
- バリデーション状態の要約
- 現在アクティブなペインや入力対象のヒント

ただし、**編集中の draft 値をフッターへ逃がさない**こと。  
編集中の値は、その場のペイン行へ直接表示する。

## バリデーション要件

バリデーションはアプリの中核機能である。

### バリデーション仕様

- **Rust の `jsonschema` crate による JSON Schema 2020-12 検証**を前提とする
- フィールドは onBlur 相当タイミングで検証する
- 保存前にも **ドキュメント全体を再検証**する
- エラーは **即時に視認可能**であること
- 可能な限り **フィールドパス単位**でエラーを表示すること
- 無効な JSON を保存しようとした場合、明示的なエラーを出すこと

### エラー表示

- エラーは隠さない
- ログだけに出して終わりにしない
- フィールド近傍か、少なくともフォームペイン内で分かるようにする
- 保存失敗・検証失敗・スキーマ未対応は区別して表示する

## JSON Schema 対応方針

### 基本方針

- **JSON Schema 2020-12** を前提とする
- ただし v1 では全機能完全対応を目指さない
- 最初に **対応サブセット** を明確に定義し、その範囲で高品質に動かす
- 未対応機能は黙って無視せず、警告または明示的な制限として扱う

### v1 で優先して対応すべき要素

- `type`
- `properties`
- `required`
- `items`
- `enum`
- `const`
- `default`
- `title`
- `description`
- `$defs`
- `$ref`（内部参照優先）
- `additionalProperties` の基本扱い

### 将来拡張として設計で意識する要素

- `oneOf`
- `anyOf`
- `allOf`
- `if` / `then` / `else`
- `dependentRequired`
- `dependentSchemas`
- `patternProperties`
- `format`
- カスタムキーワード

### 重要原則

対応していないキーワードがある場合は、以下のどれかを明示的に選ぶこと。

1. 警告して継続
2. 一部制限付きで継続
3. 処理不能として停止

**暗黙に無視してユーザーを誤認させてはならない。**

## フォーム設計方針

### 基本方針

フォームは TUI ネイティブに設計する。  
DOM 前提のフォーム抽象を無理に流用しないこと。

### 最低限必要なフィールド型

- string
- number
- integer
- boolean
- null
- object
- array

### 必須要件

- required の表示
- default の反映
- enum 選択
- nested object 対応
- array 編集
- フィールドごとの説明表示
- エラーの即時表示

### array 編集

配列編集は早い段階で設計を明示すること。少なくとも以下をサポートしたい。

- 要素追加
- 要素削除
- 要素選択
- 要素並び替え（将来拡張可）

### oneOf / anyOf の扱い

v1 で完全対応しない場合でも、将来的には **候補を選択する UI** を置けるように設計すること。

### 現在のフォーム描画方針

フォームは単なる JSON 行表示ではなく、**Web のフォームに寄せた TUI 表現**を優先する。
マウスクリックで選択できる。
編集中の値はフッターには出さず、**その場の行表示に直接反映**する。

## jq 風フィルター要件

右上ペインの JSON 出力フィルターは **jq 風** とする。

### 方針

- ユーザーは jq 風の式で出力 JSON を絞り込める
- これはあくまで **表示用フィルター** が主目的
- 元の JSON 自体を破壊的に変更しないこと

### 最低限必要な振る舞い

- フィルター文字列の入力
- フィルター評価結果の表示
- 無効なフィルター式のエラー表示
- 元 JSON とフィルター結果の区別

### 現在のフィルター実装サブセット

現状の jq 風フィルターは最小サブセットとして以下を扱う。

- `.`
- `.foo`
- `.foo.bar`
- `.array[0]`

完全な jq 互換は目指さない。

### 設計原則

- jq 実装を完全再現することを初期目標にしない
- まずは使う価値の高いサブセットから始める
- ただし UI や命名は「jq 風」であることを明示する

## 出力要件

アプリは常に現在の JSON 出力を保持・表示すること。

### 必須要件

- 現在の JSON を表示できる
- form モードでは JSON 出力パスの入力欄を持つ
- JSON を保存できる

- 保存可否の方針を UI に明示する

## 状態管理方針

状態は明示的・集約的に扱うこと。

### 推奨

- 単一の `AppState`
- action/event による遷移
- selector 的な派生データ計算
- immutable update を基本とする

### 現在の state 設計上の知見

- ペインごとのカーソル位置は `pane_cursors` としてまとめて持つ
- `schema_path` と `schema_text` は、**最後に有効だった構造化データ**とは分けて持つ
- これは、編集中に一時的に不正な JSON / パス文字列を保持するために必要
- `filter_text` と `filter_result` も分離する
- フォーム選択 index を別で持つより、`form` ペインのカーソル行を選択源にする方が整合しやすい

### 避けること

- 複数 widget 間の暗黙同期
- 場当たり的な mutable state の増殖
- 画面とロジックの状態二重管理
- バリデーション結果の散在

## 現在の実装ファイル対応

Rust 版でも、責務は概ね以下の単位に分離する。

- `domain/schema`: schema parse / ref 解決 / warning 収集
- `domain/form`: form 中間表現生成と初期値生成
- `domain/validation`: JSON Schema 検証
- `domain/filter`: jq 風フィルターの最小評価
- `app`: reducer / state / selector / JSON 更新 helper / 初期 state 構築
- `infra/fs`: UTF-8 ファイル読み書き
- `ui`: ratatui による描画とイベント制御

`ui/app.rs` に責務が寄りすぎないよう、以下は分離対象として扱う。

- キーバインド解釈
- カーソル移動
- 削除コマンド解釈
- マウスイベント解釈
- フォーム描画整形

## Definition of Done

タスク完了条件は以下。

- 期待する振る舞いが動く
- テストが追加・更新されている
- 設計が悪化していない
- ユーザーに見えるエラーやログが妥当
- 必要なら README や補助文書が更新されている

テストが通るだけでは不十分。  
見た目だけ動くのでも不十分。  
**挙動・設計・テストの三点が揃って完了**とする。
